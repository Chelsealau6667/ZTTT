## 形式发票模版管理 — 字段规则自动解析需求

### 功能说明

#### 模版管理

在「发票模版管理」页面的「模版管理」Tab 中，FSSC 发票管理员可对形式发票模版进行全生命周期管理：

- **模版列表**：展示系统内所有模版编码、名称、状态及最近更新时间，支持按状态筛选和编码/名称搜索。
- **模版导入更新**：支持单模板级别的 xlsx 文件上传，系统自动解析模版 Sheet 和字段规则配置 Sheet，识别变更内容并展示差异确认。
- **模版预览**：点击预览可查看模版渲染效果，验证字段 key 识别和取值规则绑定是否正确。
- **启用/禁用**：控制模版是否可用于发票开具流程，禁用后关联的主体配置规则同步失效。
- **模版下载**：支持下载当前版本的模版文件。

#### 主体配置

在「主体配置」Tab 中，配置开票主体在指定条件下使用的适用模版：

- **配置列表**：展示主体编码、主体名称、税率条件、金额类型及适用模版的映射关系。
- **新增配置**：选择主体、税率条件、金额类型后，系统自动级联筛选可用模版，保存后生效。
- **导入配置**：支持批量导入主体配置数据（CSV/xlsx 格式）。
- **删除配置**：移除不再使用的主体-模版匹配规则。

#### 字段规则自动解析（本期新增）

模版导入时，系统自动解析「字段规则配置」Sheet 中的规则，将可配置化的字段规则转为结构化 JSON 存入数据库，供后端渲染引擎通用处理。超出轻量引擎范围的规则标记为需硬编码，由后端专用逻辑处理。

- **解析器**：读取 Excel 字段规则 Sheet，按「取值类型」分发到不同解析策略。
- **配置化规则**（字段映射、SUM 聚合）：解析为结构化配置存入 `template_field_rule` 表。
- **硬编码规则**（条件分支、字符串拼接、流水号生成）：标记为不可配置，仅做只读展示，后端走专用函数处理。

#### 发票修改（关联功能）

在「形式发票查询及处理」页面的单据列表中，针对符合条件的记录提供「发票修改」按钮，点击后弹出编辑弹窗，支持修改已生成发票的发票号码。

### 相关规则

#### 一、字段规则解析规则

##### 解析器输入：字段规则配置 Sheet

 Sheet 每行包含以下列：

| 列名 | 说明 |
|---|---|
| 字段 key（红字内容） | Excel 模板中的红色字段标记 |
| 分类 | 申请单映射字段 / 系统生成/计算字段 / 行项目区字段 |
| 取值类型 | 字段映射 / 公式计算 / 规则计算 |
| 取值逻辑 | 自然语言描述的规则（解析器消费） |
| 说明 | 补充说明 |
| **字段编码（新增）** | 英文编码，解析器直接用作 source_field / target_field |

##### 解析器输出：`template_field_rule` 表

解析器按「取值类型」分发处理：

| 取值类型 | 解析策略 | 输出示例 | 是否可配置化 |
|---|---|---|---|
| **字段映射** | 正则匹配「申请单→xxx」或「行项目→xxx」 | `{type:"field_map", source_scope:"apply", source_field:"buyer_name"}` | ✅ 是 |
| **公式计算** | 正则匹配 SUM(行项目.xxx) | `{type:"sum", target:"line_items", field:"amount"}` | ✅ 是 |
| **规则计算** | 复杂条件分支/拼接/流水号，不匹配上述模式 | 仅标记 hardcode_reason | ❌ 否（硬编码） |

##### 解析结果统计

以「FBU-加拿大-5%GST」模版为例：

| 分类 | 数量 | 可配置化 | 需硬编码 |
|---|---|---|---|
| 申请单映射字段 | 6 条 | 4 条 | 2 条 |
| 系统生成/计算字段 | 4 条 | 4 条 | 0 条 |
| 行项目区字段 | 6 条 | 6 条 | 0 条 |
| **合计** | **16 条** | **14 条（87%）** | **2 条（13%）** |

> 注：随着各模板补充「字段编码」列并规范取值逻辑描述，可配置化比例可进一步提升。

#### 二、模版导入更新规则

1. **导入覆盖逻辑**：新模版导入后，旧版本自动归档到 `template_archive` 表，模版主表更新为最新版本。
2. **字段规则同步更新**：字段规则配置随模版导入同步更新，新增的规则插入，已存在的规则更新，`config_json` 重新解析生成。
3. **历史发票追溯**：模版更新后，已生成的历史发票仍按归档版本渲染，不受影响。

#### 三、主体配置匹配规则

系统根据以下三维度匹配适用模版：

```
entity（主体编码） + tax_rate（税率条件） + sign_type（金额类型） → template_id
```

匹配优先级：精确匹配 > 部分匹配（未指定条件项视为通配）。

#### 四、发票修改按钮展示逻辑

「发票修改」按钮的展示严格受签章文件状态与公司签章类型联合控制：

| 签章文件状态 | 公司签章类型 | 发票修改按钮展示 |
|---|---|---|
| 已上传 | —（任意类型） | **隐藏** |
| 未上传 | 电子签章 | **隐藏** |
| 未上传 | 无需签章 | **隐藏** |
| 未上传 | 线下签章 | **展示** |
| 上传异常 | —（任意类型） | **展示** |

#### 五、发票修改弹窗交互规则

1. **字段填充**：弹窗打开时自动带入当前记录的单据名称、报账单号、销方单位编码、销方单位名称及当前发票号码。
2. **编辑范围**：仅「发票号码」字段可编辑，其余字段只读展示。
3. **取消逻辑**：点击「取消」关闭弹窗，不修改任何数据。
4. **保存逻辑**：
   - 校验发票号码是否为空，为空则提示「发票号码不能为空」。
   - 触发发票号全局唯一性校验。
   - 若发票号已存在，阻断保存并提示「该发票号已存在，请更换后重试」。
   - 校验通过，更新已生成发票记录中的发票号码，关闭弹窗并刷新列表，提示「保存成功」。

#### 六、校验逻辑

- **发票号唯一性校验**：修改后保存时，系统需校验该发票号在全量发票数据中是否唯一。若存在重复，阻断保存。

### 注意事项和说明

1. **字段编码列必须补齐**：各模版维护人员在 Excel「字段规则配置」Sheet 中补填「字段编码」列，使用英文/英文点号命名（如 `buyer_name`、`line_item.amount`、`sum_line_item_amount`）。命名统一规范后，解析器才能正确映射，无需人工维护映射字典。

2. **取值逻辑描述规范化**：建议将取值逻辑统一为结构化短句，便于解析器正则匹配。例如：
   - 字段映射：`申请单→购买方名称`
   - SUM 聚合：`SUM(行项目.金额)`
   - 复杂规则保持自然语言，由硬编码函数处理

3. **硬编码规则后端统一管理**：条件分支（ITS vs FPS）、流水号生成、字符串拼接+自定义日期格式等超出轻量引擎的规则，由后端以系统内置函数方式实现（如 `generate_invoice_date()`、`generate_invoice_no()`、`calculate_due_date()`），各模板共用同一套函数族，仅入参不同。

4. **模版更新不影响历史发票**：模版导入更新后，旧版本自动归档，历史发票按归档版本回溯渲染，确保发票内容不因模版变更而变化。

5. **已上传即锁定**：签章文件状态为「已上传」时，发票号码不允许修改，「发票修改」按钮隐藏。已签署文件的完整性和不可篡改性必须保证。

6. **签章类型与主体绑定**：公司签章类型（电子签章 / 无需签章 / 线下签章）由发票主体（费用承担公司）的基础档案决定，本页面仅读取不维护。

7. **修改历史追溯**：建议在系统日志中记录发票号修改前/后值、修改人及修改时间，便于后续审计追溯。

8. **二期扩展预留**：本期采用轻量解析器，覆盖 87% 的字段规则。如果后续业务规则频繁调整，可将条件分支、字符串拼接能力升级为中等表达式引擎，届时硬编码规则可平滑迁移到配置化，数据库表结构和解析器框架无需推倒重来。
